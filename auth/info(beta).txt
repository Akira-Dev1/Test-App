***MONGO***

бд -> бд
таблица -> collection (коллекция)
строка -> document (документ)

mongosh = Mongo Shell, это CLI-клиент для MongoDB

Аналоги:
 - psql для PostgreSQL
 - mysql для MySQL

use authdb
db.users.find().pretty() - Тут я:
 - Подключился к MongoDB
 - Выбрал базу authdb
 - Посмотрел коллекцию users
 - Увидел реальные документы

Для чего нужен mongosh:
 - ручная проверка данных
 - отладка
 - тестирование схем
 - диагностика проблем

В проде mongosh не пользуются для логики, только для администрирования.


***ЗАПРОСЫ***

TCP-соединение
 └── HTTP-сессия
      ├── Строка статуса (200 OK)
      ├── Заголовки (Headers)
      └── Тело ответа (Body)  

В одном tcp-соединении может быть несколько HTTP-запросов, но все они идти будут строго по очереди, не параллельно (Если HTTP 1.1), если 2 или 3 - то могут  


***TOKENS (JWT)***

В данном проекте 3 типа токенов:
 - entry, токен входа. Передаем веб/тг клиентом в запросе /login. Дальше используется как ключ объекта (структуры) сессии. Также храним в CodeState для того, чтобы найти соответствующий AuthState на этапе подтверждения кода.
 - access, он же токен доступа. В этом проекте живет 1 минуту, содержит список разрешений, ID (скорее всего удалю его из токена). Передается через веб/тг главному модулю для реализации основного функционала приложения
 - refresh, токен обновления. В этом проекте живет 7 дней, хранится в Mongo (сейчас хранятся в списке в документе-пользователе) используется для обновления access токена и для /logout (как я понял, будет уточнено)
